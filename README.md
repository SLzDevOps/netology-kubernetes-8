# Задача 1
## Расчет ресурсов для Kubernetes-кластера

### 1. Потребности приложения

| Компонент | Копий | ОЗУ (всего) | CPU (всего) |
| :--- | :---: | :---: | :---: |
| База данных | 3 | 12 ГБ | 3 ядра |
| Кеш | 3 | 12 ГБ | 3 ядра |
| Фронтенд | 5 | 0.25 ГБ | 1 ядро |
| Бекенд | 10 | 6 ГБ | 10 ядер |
| **ИТОГО** | **21** | **30.25 ГБ** | **17 ядер** |

### 2. Расчет нод
#### Минимальная конфигурация (N+1)
*   **Количество нод:** 3
*   **Параметры ноды:** 12 vCPU, 16 ГБ RAM
*   **Всего в кластере:** 36 vCPU, 48 ГБ RAM

#### Рекомендуемая конфигурация (С учетом служебных ресурсов и запаса)
*   **Количество нод:** 4
*   **Параметры ноды:** 8 vCPU, 16 ГБ RAM
*   **Всего в кластере:** 32 vCPU, 64 ГБ RAM

### 3. Обоснование
*   **Отказоустойчивость:** Выбранное количество нод (3 или 4) позволяет распределить 3 реплики базы данных и 3 реплики кеша строго на разные физические сервера (через правила PodAntiAffinity), гарантируя доступность сервисов.
*   **Служебные ресурсы:** На каждой ноде заложено ~1 vCPU и ~2 ГБ RAM под накладные расходы (ОС, Kubelet, Container Runtime, системные демоны), что уменьшает полезную (Allocatable) емкость сервера.
*   **Запас на отказ:** В рекомендуемой конфигурации (4 ноды по 8 CPU / 16 RAM) при выходе из строя одной любой ноды оставшиеся 3 ноды суммарно сохраняют 21 vCPU и 42 ГБ полезной емкости. Этого полностью хватает для бесперебойной работы приложения, требующего 17 ядер и 30.25 ГБ ОЗУ.
